# 变体管理最佳实践 (Variant Management Best Practices)

随着项目的快速迭代，由于业务决策、网络环境和基础设施的差异，我们的项目中同时存在着多条平行发展的逻辑（变体 Variant）。为了避免代码日益臃肿，我们需要对不同类型的变体进行分类，并采用与之匹配的最佳实践。

以下是针对我们目前的三种典型变体场景的解决方案：

## 1. 业务功能分支（短期变体/A-B 测试）
**场景：** 小程序端“上传音频” vs “联网自动语音检测”。两者功能重叠，最终只会留存其一。
**最佳实践：特性开关 (Feature Toggles)**
- **方案描述：** 在代码库中保留这两套代码，通过一个全局配置对象（或环境变量）来动态决定启用哪个。例如在代码的入口或单独的配置文件 `config/features.js` 中设置 `enableAutoSpeechDetection = true/false`。
- **实施原则：** 
  - 减少耦合：尽量将两套不同的逻辑封装为单独的类、方法或组件，不要在核心业务代码里到处散落 `if-else`。
  - **最重要的一点（防腐败）：** 这种变体具有**时效性**。一旦产品确定了最终保留的方案，必须在下一次重大的代码合并中彻底删除落选方案的废弃代码以及该特性开关，消除技术债。

## 2. 部署拓扑结构差异（长期变体/环境配置）
**场景：** 服务端“国内部署” vs “海外部署”。国内需要额外的微服务连接 Google 服务，海外则不必。
**最佳实践：容器编排 Profiles + 环境变量注入**
- **方案描述：** 不要在主从代码中编写诸如 `if (isCN) { startProxyServer(); }` 的逻辑，服务启动和依赖管理应该交给外部编排工具（如 Docker-Compose / Kubernetes）。
- **实施原则：**
  - **网络层：** 无论是微服务还是 Google 直连，都抽象为一个共同的 API 端点。主服务端根据注入的 `GOOGLE_SERVICE_URL` 环境变量去建立连接。
  - **编排层：** 使用 Docker Compose 的 `profiles` 或不同环境的部署脚本。当执行国内部署时，启动主服务容器以及 Google Proxy 微服务容器，并将主服务的 `GOOGLE_SERVICE_URL` 指向该微服务；海外部署时，只启动主服务容器，将 URL 直接指向 Google 官方端点。这样可以杜绝资源冗余。

## 3. 基础设施 / 第三方云厂商替换（长期变体/架构演进）
**场景：** 服务端“微信托管云” vs “自建服务器”。目前依赖微信云开发（CloudBase），但不排除未来自主部署。
**最佳实践：六边形架构 (Hexagonal Architecture) / 依赖注入与端口适配器模式**
- **方案描述：** 坚决执行“控制反转”（IoC）。业务层代码不能感知到底层是在使用微信特有的 API 还是在用自建方案。
- **实施原则：**
  - **抽象接口 (Ports)：** 定义 `IUserStorage`、`IAudioUploader` 这样的上层接口。
  - **分离实现 (Adapters)：** 创建 `WechatCloudStorageAdapter`（里面封装了诸如 `cloud.uploadFile` 等特定微信 API）与 `SelfHostedS3Adapter`（里面封装了 AWS/MinIO 的具体行为）。
  - **运行时注入：** 服务引导阶段（Bootstrapping），读取环境变量 `INFRA_PROVIDER=WX_CLOUD`。根据此变量，将 `WechatCloudStorageAdapter` 实例注入给所有的核心业务模块。
  - **收益：** 将来迁移到自建云时，核心业务逻辑 `0` 修改，只需要新增一个 Adapter 并更改该环境变量即可平滑过渡。
